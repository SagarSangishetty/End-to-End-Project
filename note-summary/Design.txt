Key design decisions and why:
X-Request-ID as the correlation ID вҖ” Frontend generates a UUID on every request. Passes it as an HTTP header to every service call. Every service logs it, includes it in traces, and passes it downstream. One request ID traces the full journey: frontend вҶ’ user service вҶ’ order service вҶ’ database. In Jaeger you search by this ID and see the entire chain.
One database, three schemas вҖ” For simplicity and cost in a learning environment. In real production each microservice would have its own database. You mention this in interviews: "For this project I used a shared PostgreSQL with separate schemas per service. In a real production environment each service would own its own database to maintain true independence." Shows maturity.
Two namespaces on EKS вҖ” ecommerce-dev and ecommerce-prod. Same cluster, isolated environments. Same pattern you already used in your observability project.
Terraform for infrastructure вҖ” VPC and EKS provisioned via your existing modules. Not manual clicking in console.

Build order вҖ” what we do first to last:
Step 1 вҶ’ Create GitHub repo, branching strategy, folder structure
Step 2 вҶ’ Write application code (simple CRUD вҖ” 1 day max)
Step 3 вҶ’ Dockerise all 4 services
Step 4 вҶ’ Test locally with docker-compose
Step 5 вҶ’ Write Kubernetes manifests
Step 6 вҶ’ Deploy to EKS manually first (confirm it works)
Step 7 вҶ’ Add CI/CD pipelines (GitHub Actions)
Step 8 вҶ’ Add observability stack
Step 9 вҶ’ Configure alerts
Step 10 вҶ’ Test end to end вҖ” trace one request through everything
Each step is completable independently. We don't move to next until current step is solid
